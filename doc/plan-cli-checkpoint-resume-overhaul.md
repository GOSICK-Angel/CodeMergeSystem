## 1. Background & Motivation

### Current Pain Points

1. **CLI flow fragmented**: User must run `merge init` -> `merge validate` -> `merge run --config xxx.yaml` across 3 separate steps. Config file path must be manually managed.

2. **Checkpoint spam**: A single merge run generates 50+ checkpoint files (`phase2_batch_10`, `phase3_file_x`, `phase5_repair_0`, ...). These intermediate checkpoints provide minimal value beyond debug tracing.

3. **Resume is broken**: `_file_diffs`, `_upstream_commits`, `_replayable_commits`, `_non_replayable_commits` are stored as private attributes via `object.__setattr__()` and **never serialized** to checkpoint JSON. On resume, all downstream phases silently fail or produce empty results.

4. **Output location confusing**: All merge artifacts (`outputs/`, `MERGE_RECORD/`) are written relative to CodeMergeSystem's CWD, not the target project. When CodeMergeSystem is pip-installed and used in another project, output paths become meaningless.

## 2. Dev Mode vs Production Mode

### Why Distinguish?

| Scenario | CWD | Expected behavior |
|----------|-----|-------------------|
| **Development**: developer working on CodeMergeSystem itself | `CodeMergeSystem/` | Logs, traces, checkpoints all in `./outputs/debug/` for easy inspection |
| **Production**: user pip-installed CodeMergeSystem, runs `merge` in their project | `their-project/` | Merge artifacts in `their-project/.merge/`, system logs in a well-known system path |

### Detection Strategy

```python
def is_dev_mode() -> bool:
    """Detect if running from source (dev) vs pip-installed (prod)."""
    # Method 1: Check env var override
    if os.environ.get("MERGE_DEV") == "1":
        return True
    # Method 2: Check if running from a directory containing pyproject.toml
    #           with [project] name = "code-merge-system"
    pkg_root = Path(__file__).resolve().parents[2]
    pyproject = pkg_root / "pyproject.toml"
    return pyproject.exists() and "code-merge-system" in pyproject.read_text()
```

### Behavior Matrix

| Artifact | Dev Mode | Production Mode |
|----------|----------|-----------------|
| Merge config | `./merge-config.yaml` (legacy) | `<target>/.merge/config.yaml` |
| API keys | `./.env` (legacy) | `<target>/.merge/.env` + `~/.config/code-merge-system/.env` (fallback) |
| Checkpoint | `./outputs/debug/checkpoints/` | `<target>/.merge/runs/<run_id>/checkpoint.json` |
| Reports | `./outputs/` | `<target>/.merge/runs/<run_id>/` |
| MERGE_PLAN | `<target>/MERGE_RECORD/` | `<target>/.merge/plans/` |
| Execution log | `./outputs/debug/run_<id>.log` | `~/.local/share/code-merge-system/logs/run_<id>.log` |
| LLM traces | `./outputs/debug/llm_traces_<id>.jsonl` | `~/.local/share/code-merge-system/logs/llm_traces_<id>.jsonl` |
| Structured log | `./outputs/debug/run_<id>.jsonl` | `~/.local/share/code-merge-system/logs/run_<id>.jsonl` |

> **Note**: Dev mode keeps full backward compatibility. Existing `./outputs/` paths continue to work.

### System Log Path Resolution

```python
import platformdirs

def get_system_log_dir() -> Path:
    """Get system-level log directory for CodeMergeSystem."""
    if is_dev_mode():
        return Path("./outputs/debug")
    return Path(platformdirs.user_data_dir("code-merge-system")) / "logs"
```

## 5. Issue 2: Checkpoint Cleanup

### Current Problem

Each phase/batch/file produces a separate checkpoint file:
```
run_abc123_after_phase1.json        # ~50KB
run_abc123_after_phase1_5.json      # ~80KB
run_abc123_phase2_batch_10.json     # ~100KB
run_abc123_phase2_batch_20.json     # ~120KB
run_abc123_phase3_auth_py.json      # ~150KB
run_abc123_phase3_config_ts.json    # ~160KB
run_abc123_phase5_repair_0.json     # ~200KB
run_abc123_awaiting_human.json      # ~220KB
run_abc123_completed.json           # ~250KB
run_abc123_latest.json              # -> symlink
```

A 100-file merge can produce 50+ checkpoint files totaling 5-10MB.

### New Strategy: Single Rolling Checkpoint

**Default mode**: Only keep ONE checkpoint file per run (`checkpoint.json`), overwritten on each save.

**Debug mode** (`--debug` flag or `output.debug_checkpoints: true`): Keep all checkpoints with tags (current behavior).

### Implementation

#### 5.1 Checkpoint Class Changes

**File**: `src/core/checkpoint.py`

```python
class Checkpoint:
    def __init__(self, run_dir: Path, *, debug_checkpoints: bool = False):
        self.run_dir = run_dir
        self.run_dir.mkdir(parents=True, exist_ok=True)
        self._debug = debug_checkpoints
        if self._debug:
            self._debug_dir = run_dir / "checkpoints_debug"
            self._debug_dir.mkdir(exist_ok=True)

    def save(self, state: MergeState, tag: str = "") -> Path:
        # Always overwrite the single checkpoint
        main_path = self.run_dir / "checkpoint.json"
        data = state.model_dump(mode="json")
        _atomic_write(main_path, data)
        state.checkpoint_path = str(main_path)

        # In debug mode, also save tagged copy
        if self._debug and tag:
            debug_path = self._debug_dir / f"{tag}.json"
            debug_path.write_text(
                json.dumps(data, indent=2, default=str), encoding="utf-8"
            )

        return main_path

    def load(self, checkpoint_path: Path | None = None) -> MergeState:
        path = checkpoint_path or (self.run_dir / "checkpoint.json")
        if not path.exists():
            raise FileNotFoundError(f"Checkpoint not found: {path}")
        raw = path.read_text(encoding="utf-8")
        data = json.loads(raw)
        return MergeState.model_validate(data)
```

#### 5.2 Atomic Write (Prevent Corruption)

```python
def _atomic_write(path: Path, data: dict) -> None:
    """Write JSON atomically via temp file + rename."""
    tmp = path.with_suffix(".tmp")
    tmp.write_text(json.dumps(data, indent=2, default=str), encoding="utf-8")
    tmp.rename(path)  # atomic on POSIX
```

This also fixes the corruption risk identified in the resume bug analysis.

#### 5.3 Config Addition

**File**: `src/models/config.py` — `OutputConfig`

```python
class OutputConfig(BaseModel):
    ...
    debug_checkpoints: bool = False  # Keep all tagged checkpoints
```

#### 5.4 Remove Symlink Logic

The `_latest` symlink mechanism is no longer needed. The single `checkpoint.json` is always the latest. Remove `get_latest()`, `list_checkpoints()` symlink handling.

## 6. Issue 3: Resume Bug Fix

### Root Cause Analysis

**CRITICAL BUG**: Private state not serialized.

`InitializePhase._run_sync()` stores computed data as private attributes:

```python
# src/core/phases/initialize.py:179-192
object.__setattr__(state, "_file_diffs", file_diffs)
object.__setattr__(state, "_upstream_commits", upstream_commits)
object.__setattr__(state, "_replayable_commits", replayable)
object.__setattr__(state, "_non_replayable_commits", non_replayable)
```

These bypass Pydantic's field system, so `model_dump()` never includes them. On resume, all downstream phases see `None` / `[]`.

### Fix Strategy

#### 6.1 Promote Private Attributes to Pydantic Fields

**File**: `src/models/state.py`

Add proper fields to `MergeState`:

```python
from src.models.diff import FileDiff

class MergeState(BaseModel):
    ...
    # Previously private attrs — now serialized for checkpoint resume
    file_diffs: list[FileDiff] = Field(default_factory=list)
    upstream_commits: list[dict[str, Any]] = Field(default_factory=list)
    replayable_commits: list[dict[str, Any]] = Field(default_factory=list)
    non_replayable_commits: list[dict[str, Any]] = Field(default_factory=list)
```

#### 6.2 Update InitializePhase

**File**: `src/core/phases/initialize.py`

Replace:
```python
object.__setattr__(state, "_file_diffs", file_diffs)
object.__setattr__(state, "_upstream_commits", upstream_commits)
object.__setattr__(state, "_replayable_commits", replayable)
object.__setattr__(state, "_non_replayable_commits", non_replayable)
```

With:
```python
state.file_diffs = file_diffs
state.upstream_commits = upstream_commits
state.replayable_commits = replayable
state.non_replayable_commits = non_replayable
```

#### 6.3 Update All Consumers

Replace every `getattr(state, "_file_diffs", None)` with `state.file_diffs`.

Affected files (grep `_file_diffs|_upstream_commits|_replayable|_non_replayable`):
- `src/core/phases/initialize.py` (producer)
- `src/agents/planner_agent.py` (consumer)
- `src/core/phases/auto_merge.py` (consumer)
- `src/core/phases/conflict_analysis.py` (consumer)
- `src/tools/merge_plan_report.py` (consumer)

#### 6.6 Checkpoint Integrity Validation

**File**: `src/core/checkpoint.py`

```python
def load(self, checkpoint_path: Path | None = None) -> MergeState:
    path = checkpoint_path or (self.run_dir / "checkpoint.json")
    if not path.exists():
        raise FileNotFoundError(f"Checkpoint not found: {path}")
    try:
        raw = path.read_text(encoding="utf-8")
        data = json.loads(raw)
        return MergeState.model_validate(data)
    except json.JSONDecodeError as e:
        raise RuntimeError(
            f"Checkpoint corrupted (invalid JSON): {path}\n{e}"
        ) from e
    except ValidationError as e:
        raise RuntimeError(
            f"Checkpoint schema mismatch (possibly from older version): {path}\n{e}"
        ) from e
```

## Appendix: Key Decision Records

### ADR-1: Why `.merge/` not `.code-merge/`?

Short, memorable, matches the tool's purpose. Similar to `.git/`, `.github/`, `.vscode/`. Low collision risk with existing tooling.

### ADR-2: Why single checkpoint, not LRU(N)?

Simplicity. The only use case for resume is "continue from where I left off", which only needs the latest state. Debug checkpoints serve a different purpose (tracing phase-by-phase evolution) and are opt-in via `--debug`.

### ADR-3: Why `platformdirs` for system paths?

Cross-platform standard (`~/.local/share/` on Linux, `~/Library/Application Support/` on macOS, `%APPDATA%` on Windows). Avoids reinventing path discovery. Lightweight dependency (zero transitive deps).

### ADR-4: Why not re-run InitializePhase on resume?

InitializePhase performs expensive git operations (diff computation, file classification, commit enumeration). Re-running it:
1. Wastes 5-30 seconds on large repos
2. May produce different results if the repo state changed since the original run
3. Breaks the principle of "resume from exact state"

Proper serialization of computed data is the correct fix.
