你是一个资深的软件工程架构师 + 多智能体系统设计者 + 代码合并专家。请你为我**设计并实现一个可运行的 Multi-agent Code Merge System**，用于解决“长期分叉分支与上游持续演进分支之间无法直接 git merge”的复杂代码合并问题。

## 一、问题背景

在真实项目中，用户通常会基于某个历史版本进行深度二次开发；与此同时，原始项目（上游仓库）会持续迭代，新增功能、修复 bug、做性能优化、调整架构。经过较长时间后，用户自己的分支与上游目标分支之间会出现大量差异和高冲突，导致：

* 无法直接依赖 git merge 自动完成合并
* 即使能合并，也可能丢失一方的重要功能 / 优化 / bugfix
* 当同一文件、同一逻辑区域被两个分支同时修改时，仅靠文本冲突无法得出正确结果
* 某些冲突需要追溯“为什么改”而不仅仅是“改了什么”

我需要一个多智能体系统，帮助用户在这种背景下**自主、系统化、可审计地完成代码合并**。

---

## 二、系统目标

请实现一个 Multi-agent 系统，目标如下：

1. **完整保留当前分支（Current / Custom Branch）中的定制功能、优化和 bugfix**
2. **完整引入目标合入分支（Target / Upstream Branch）中的功能、优化和 bugfix**
3. 当两个分支对同一文件或同一逻辑区域都有修改时，不是简单做文本拼接，而是：

   * 识别冲突点
   * 分析双方修改内容
   * 追溯修改原因（intent / rationale）
   * 基于语义和上下文决策如何合并
4. 对于系统无法高置信度决策的冲突，**绝不能忽略、跳过或擅自拍板**

   * 必须进入 human-in-the-loop 流程
   * 或沉淀到统一的决策文档中，供人类集中裁决
5. 整个系统必须具备：

   * 明确的 agent 分工
   * 清晰的状态流转
   * 可追溯的决策依据
   * 可重复执行的流程
   * 审查与执行相分离

---

## 三、核心设计原则

请在实现中严格遵守以下原则：

### 1. 不丢失原则

任何一方分支中已有的功能、优化、bugfix，都不能因为合并流程而被静默丢失。

### 2. 语义优先原则

不能只基于文本 diff 做机械合并，应尽可能基于以下信息进行判断：

* git diff
* git blame
* commit message
* 相关上下文代码
* 调用链 / 依赖关系
* 测试 / 类型检查 / lint / 构建结果
* 注释、文档、配置、接口定义变化

### 3. 可解释原则

每一个冲突点最终为什么这样合并，系统都需要给出解释：

* 当前分支改了什么
* 目标分支改了什么
* 双方各自想解决什么问题
* 最终采用了哪种合并策略
* 为什么这样做更合理

### 4. 不确定即升级原则

只要系统无法高置信度判断，就必须显式升级，不允许：

* 忽略冲突
* 默认保留某一方
* 用模糊描述掩盖未解决问题

### 5. 审查隔离原则

执行合并的 agent 不能同时作为最终裁决者，必须有独立的审查 / 评估角色。

## 四、你需要实现的系统能力

请实现一个**真正可运行**的系统，而不是只给概念设计。至少包含以下能力：

### A. 仓库分析能力

* 识别当前分支、目标分支、merge base
* 统计文件级差异
* 对文件分类，例如：

  * 仅上游修改
  * 仅当前分支修改
  * 双方都修改
  * 上游新增 / 当前缺失
  * 当前新增 / 上游无此文件
  * 删除 / 重命名 / 移动文件
* 对高风险文件进行标记，例如：

  * 鉴权 / 登录 / 权限
  * 配置 / 环境变量
  * 数据模型 / schema
  * API 契约
  * 核心业务逻辑
  * 并发 / 异步 / 任务系统
  * 构建 / 部署 / 启动入口

### B. 冲突理解能力

对“双方都修改”的文件，不能止步于 diff，至少要进一步分析：

* 改动属于什么类型：

  * bugfix
  * 重构
  * 功能增强
  * 性能优化
  * 安全修复
  * 风格整理
  * 删除废弃逻辑
* 改动发生在什么语义单元：

  * 函数 / 方法
  * 类
  * 配置项
  * 类型定义
  * SQL / schema
  * 接口协议
* 修改是否互斥、兼容、互补，还是存在隐藏风险

### C. 原因追溯能力

系统需要尽量追溯“为什么改”，包括但不限于：

* 相关 commit message
* git blame / 历史作者信息
* 邻近提交
* issue / PR 信息（如果可获得）
* 注释 / 文档 / 测试变化
* 从代码上下文推断修改意图

### D. 合并决策能力

对于每个冲突点，系统应能输出以下几类决策之一：

1. **直接采用当前分支**
2. **直接采用目标分支**
3. **语义融合两方改动**
4. **保留一方并手动补入另一方关键逻辑**
5. **无法自动决策，升级给人类**

### E. 审查与门禁能力

合并之后，系统必须执行审查和基础门禁，例如：

* lint
* type-check
* unit test
* build
* 关键文件 diff 审查
* upstream 特性是否确实被引入
* current 定制逻辑是否仍然存在
* 是否出现 silent regression 风险

### F. Human-in-the-loop 能力

对于无法高置信度处理的问题，系统必须：

* 生成结构化冲突报告
* 提供候选方案
* 给出每个方案的利弊
* 标记需要人类补充的关键信息
* 在人类裁决后，能够继续执行后续流程

## 五、建议的 Multi-agent 架构

请至少设计并实现以下 Agent，必要时你也可以补充更多 Agent，但必须说明职责边界。

### 1. Planner Agent

负责：

* 理解合并任务
* 分析仓库总体差异
* 生成阶段化合并计划
* 识别高风险区域
* 决定哪些任务可以自动执行，哪些需要重点审查

输出：

* merge plan
* phase 划分
* 风险清单
* 优先级建议

### 2. Executor Agent

负责：

* 按计划逐阶段执行合并
* 对具体文件实施合并动作
* 调用 git / diff / blame / 测试等能力
* 写出中间结果
* 对每个文件记录处理日志

要求：

* 不得在高不确定性场景下私自决策
* 不得跳过冲突不记录

### 3. Judge / Reviewer Agent

负责：

* 独立审查 Executor 的结果
* 验证是否遗漏上游逻辑
* 验证是否破坏当前分支已有逻辑
* 对结果给出 PASS / CONDITIONAL / FAIL
* 当失败时给出明确修复指令，而不是泛泛而谈

### 4. Conflict Analyst Agent

负责：

* 深入分析复杂冲突
* 对冲突进行语义拆解
* 追溯双方修改原因
* 为 Executor / Judge / Human 提供决策依据

### 5. Human Interface Agent

负责：

* 生成给人类看的决策文档
* 汇总待裁决事项
* 将技术细节翻译成可读的选择题
* 接收人类裁决并反馈给系统

## 六、系统输入

请设计系统支持以下输入：

* repository path
* current branch
* target branch
* optional merge base
* optional file scope / directory scope
* optional custom preservation rules
* optional critical paths / forbidden overwrite paths
* optional test commands / build commands / lint commands
* optional review policy
* optional human decision file

示例：

```yaml
repo_path: /path/to/repo
current_branch: custom/main
target_branch: upstream/v1.14.0
merge_base: null
scope:
  - api/
  - web/
preserve_rules:
  - "必须保留当前分支中的 SSO 逻辑"
  - "必须保留当前分支中的监控埋点"
  - "必须引入目标分支中的新插件机制"
quality_gates:
  - "pnpm type-check"
  - "pnpm test"
  - "pnpm build"
  - "pytest"
review_policy:
  require_judge_for_all_conflicted_files: true
  require_human_for_high_risk_conflicts: true
```

## 七、系统输出

请确保系统最终至少输出以下产物：

### 1. Merge Plan

例如：

* Phase 1：仓库分析与分类
* Phase 2：低风险自动合并
* Phase 3：高风险冲突分析
* Phase 4：人工决策汇总
* Phase 5：审查与门禁
* Phase 6：最终报告

### 2. File-level Merge Decisions

每个文件都要有记录：

* 文件路径
* 分类
* 冲突级别
* 采用的策略
* 原因说明
* 是否需要人类参与
* 最终状态

### 3. Human Decision Report

对无法决策的问题输出统一文档，例如：

* 冲突背景
* 双方改动摘要
* 风险说明
* 候选方案 A / B / C
* 推荐方案
* 需要人类回答的问题

### 4. Judge Report

对每个阶段或每批文件输出：

* PASS / CONDITIONAL / FAIL
* 问题清单
* veto 条件
* 修复建议

### 5. Final Merge Summary

最终总结：

* 合并了哪些上游能力
* 保留了哪些当前分支能力
* 哪些问题仍待人工确认
* 门禁结果
* 风险提示

## 八、关键行为约束

请在实现中显式编码以下约束：

1. **禁止 silent conflict resolution**

   * 任何未解决冲突都必须被记录

2. **禁止默认“目标分支优先”或“当前分支优先”**

   * 必须按语义决策

3. **高风险文件必须进入更严格审查**

   * 至少需要 Conflict Analyst + Judge 参与

4. **Judge 必须是只读角色**

   * 不直接修改代码，只负责审查和给出修复意见

5. **Executor 修改后必须可回溯**

   * 每一步变更要有日志或决策记录

6. **无法判断时必须升级**

   * 输出结构化信息给人类，而不是停止在“存在冲突”

## 十、验收标准

你实现的系统至少要满足以下验收标准：

* 能识别两分支差异并分类
